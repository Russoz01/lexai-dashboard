### [P0-2] FOUC em produção: stylesheet carrega antes do boot script de tema

**Where:** `src/app/layout.tsx:184-197` (boot script) e ordem de render do Next.js head

**What:** No HTML server-rendered, `<link rel="stylesheet" href="/_next/static/css/app/layout.css">` aparece em posição 3 do `<head>`, ANTES do boot script `<script>` que seta `data-theme`. Confirmado via `curl localhost:3000`:

```html
<head>
  <meta charSet="utf-8"/>
  <meta name="viewport" .../>
  <link rel="stylesheet" href="/_next/static/css/app/layout.css?..."/>  <!-- carrega aqui -->
  <link rel="preload" as="script" .../>
  <script src=".../main-app.js" async></script>
  ...
  <title>...</title>
  ...
  <script>(function(){ ... })()</script>  <!-- boot script aparece DEPOIS -->
```

CSS `:root,[data-theme="light"]` em globals.css linha 14-16 casa quando `<html>` ainda não tem `data-theme` (default). Resultado: browser pinta cream antes do JS aplicar `data-theme="dark"` = flash creme→preto pra usuário default-dark.

**Why bad:** Flash visível em primeiro paint, especialmente perceptível em conexões mobile lentas que era exatamente o cenário de demo.

**Fix prompt:**
```
Trocar a regra base do globals.css de
  :root, [data-theme="light"] {
para
  [data-theme="light"] {

E adicionar um bloco de fallback ANTES dele:
  :root { /* default tokens dark, idênticos a [data-theme="dark"] */ }

Assim a SSR sem data-theme renderiza dark (default) e a primeira tinta já é
correta, antes mesmo do JS executar. O boot script depois ratifica.

Bonus: usar next/script com strategy="beforeInteractive" não ajuda pra isso
porque o problema é precedência DOM order, não execution.
```

**Effort:** S (15 minutos — basta separar o `:root` block do `[data-theme="light"]`)

## Light mode specific

### What works (confirmado)
- `data-theme="light"` aplicado corretamente no `<html>` (verificado runtime)
- Tokens CSS trocam corretos: `--bg-base: #F5EFE6`, `--text-primary: #132025`
- LocalStorage persiste preferência (`pralvex-theme: light` survives reload)
- Cross-tab sync funciona (storage event listener em ThemeContext linha 142)
- prefers-color-scheme listener engaja quando pref='system' (linha 124)
- Boot script no `<head>` executa e seta data-theme antes do hydration
- Toggle UI: pill segmentado com 3 estados visualmente claros
- Toggle responde rápido (5 cliques sequenciais sem race condition)

### What needs fix (the gap)
- **Bug semântico:** 16 rotas usam `bg-black` Tailwind, ignoram tokens. User clica light → não vê light.
- **FOUC potencial:** boot script renderiza após `<link rel="stylesheet">` no SSR.
- **Toggle ausente:** 5 páginas não têm ThemeToggle (privacidade, termos, login, reset-password, not-found).
- **Cookie consent não-tematizado:** cores hardcoded em CookieConsent.tsx.
- **Stale comment:** dashboard/layout.tsx:18 afirma "Light mode foi removido".

### Suggested improvements
- Criar utility classes `.surface-base`, `.surface-card`, `.text-on-surface` em globals.css que usam vars — mais limpo que inline styles.
- Considerar adicionar 4 estados ao toggle: light / dark / system / sepia (atelier paper). Pode ser branding amigável pra advogado que lê 6h por dia.
- Pequena animação de transição CSS quando data-theme muda (já existe `transition: background-color 0.2s` em html — falta replicar em surfaces).

## Recommendations (top 3)

### 1. Tokenizar `bg-black text-white` em todas as rotas públicas (P0-1)
Maior impacto único. Sem isso o feature de light mode é placebo. Sugestão: criar PR isolado tocando apenas as 16 páginas listadas, com search-replace metódico, antes de qualquer outro polish. **~2-3 horas de trabalho**.

### 2. Fix sitemap legacy + boot script ordering (P0-2, P0-3, P0-4)
Mudanças de SEO + first-paint que pegam o demo desavisado. **~30 minutos total**.

### 3. Remover wrapper visibility:hidden do ThemeProvider (P1-6)
Quick win pra Web Vitals. CLS impact óbvio em qualquer Lighthouse mobile run. **5 minutos**.
